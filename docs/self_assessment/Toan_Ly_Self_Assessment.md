# Self Assessment

## Part A - Individual Contribution

Throughout this project, my main contribution was building the core backend system for our AI mental health assistant. I was responsible for designing and implementing the RAG (Retrieval Augmented Generation) pipeline using DSM5 as the knowledge base, which allowed the system to generate more grounded and relevant responses. In addition, I developed key backend features such as user authentication, persistent chat memory, and mood tracking system that records and visualizes user data over time. I also built API endpoints using FastAPI and connected them to a PostgreSQL database, ensuring that all components, from user login to chat interactions, worked together as a cohesive system. Compared to my initial assessment last Fall semester, I believe I significantly expanded my skills in backend engineering and integrating AI LLM and RAG into real applications.

Through this work, I learned how to design a fullstack AI system beyond just model experimentation. One of my biggest successes was getting all components such as RAG pipeline, database, APIs, and frontend, to work together in a stable workflow. However, I also faced several challenges, especially with dependency conflicts, Docker setup, and integrating external services like Qdrant and PostgreSQL. Debugging these issues required a lot of persistence and problem solving. These obstacles helped me build stronger skills in debugging, system integration, and deployment. 

## Part B - Team Assessment

As a team, we successfully build an end-to-end AI mental health assistant that allows users to interact with a chatbot, track their mental health status, and review historical data. We were able to combine different components, such as frontend UI, backend APIs, AI pipeline, and database, into a single working system. Our project demonstrated not only technical functionality but also a meaningful application in mental health support. Throughout this process, I learned that effective teamwork requires responsibilities, consistent communication, and flexibility when plans change. Our team worked well in dividing tasks based on each teammate's strengths, which helped us make steady progress throughout the semester.

However, not everything of teamwork was perfect. At times, integration between components became challenging due to differences in development pace and assumptions about how different parts of the system would connect. Even though both team members put in the same amount of effort, we sometimes worked independently for too long, which led to mismatches when we tried to combine our work. As a result, we had to spend extra time debugging and aligning our implementations. This experience showed us that equal effort does not always guarantee smooth collaboration, especially in a technical project where components are highly interdepedent. Moving forward, I learned that more frequent communication, clearer expectations, and earlier integration testing would have helped us avoid these issues and work more effective as a team.
